# AZ.0 — JSON schema for `_semantic_locks/<instance>.<dialect>.json`

**Phase**: AZ — retire byte-locked seeds → semantic locks
**Status**: spike + design lock
**Date**: 2026-05-23
**Owner**: AY ⇒ AZ track

---

## Context

Phase AY shipped (v11.15.0): production seed flows through the spine
pipeline; every row carries `metadata.scenario_id`. The byte-locked
seed files (`tests/data/_locked_seeds/<instance>.<dialect>.sql`,
~9.4 MB × 3 dialects = ~28 MB total) gate CI on per-line SQL byte
equality. That gate has gotten in the way:

- AY needed to re-lock TWICE for documented drift (per-row INSERT
  syntax shift + metadata payload additions). Each re-lock is a
  28 MB diff to review.
- The bytes encode SQL formatting decisions (column order,
  whitespace, comment headers) that aren't load-bearing. Two
  generator implementations that produce the SAME violation set
  but different SQL bytes both fail the byte-lock gate.
- The bytes don't expose what they actually test: "does this
  scenario produce the right invariant violations?" is what we
  care about; the SQL text is implementation.

Phase AZ replaces the byte locks with **semantic locks**: a
per-`(instance, dialect)` JSON file capturing the **violation set
the detectors produce post-emit**. Implementation churn that
preserves violations passes the gate; byte-locked breaks. The lock
guards INTENT, not implementation.

The semantic-lock primitive already exists on the spine
(`common/spine/semantic_lock.py::semantic_lock` returns
`dict[str, frozenset[Violation]]`). AZ.0's job: lock the JSON
serialization shape, AZ.1+ build the CLI + tests + gate swap.

---

## Inputs to the lock

`semantic_lock(conn, ALL_INVARIANTS)` returns:

```python
{
    "drift": frozenset({
        RuleViolation(invariant="drift", identity=frozenset({
            ("account_id", "acct-drift-child-CustomerSubledger"),
            ("business_day", date(2030, 1, 2)),
            ("drift", 5.0),
        })),
        ...
    }),
    "overdraft": frozenset(...),
    ...
}
```

Each `Violation` is one of three subtypes from AY.2.a:

- `RuleViolation` — matview-detected rule break
- `CoverageObservation` — seed-color presence claim
- `AuditFixture` — audit-PDF input marker

Subtype matters for equality (`__eq__` checks `__class__`), so the
JSON shape must preserve it.

`identity` is a `frozenset[tuple[str, object]]` — JSON needs sorted
key:value mapping. Values are typically `str` / `int` / `float` /
`date`.

## JSON shape (locked)

### File: `tests/data/_semantic_locks/<instance>.<dialect>.json`

```json
{

  "scenario_fingerprint": {
    "instance": "spec_example",
```

### Key decisions

1. **`_comment` field at top-level.** JSON doesn't support
   comments; embedding a sentinel `_comment` key gives reviewers
   context when they open the file. The loader ignores it (any
   `_`-prefixed top-level key is ignored, leaving room for future
   `_metadata` / `_provenance` fields without a schema bump).

2. **`scenario_fingerprint`** for early-mismatch detection. If the
   file's `instance` or `dialect` doesn't match the test's
   parameters, the gate fails with a clear "file is from the wrong
   shape" error rather than a violation-set diff. `canonical_anchor`
   pins the date the lock was generated against. `schema_version`
   bumps when the JSON serialization shape changes (a `dict` vs
   `list` flip on `identity`, or a new mandatory top-level key);
   loaders gate on version match.

3. **`violations` is a dict keyed by `invariant.name`**, sorted by
   key in the rendered JSON. Values are LISTS (not sets) sorted by
   `(kind, identity-tuple-sorted-repr)` so re-runs are byte-stable.
   Empty invariants ARE included as empty arrays (`"drift": []`) so
   diffing two locks shows when an invariant moved from
   has-violations to empty.

4. **Per-violation `{kind, identity}` shape.** `kind` is one of
   `"rule_violation"` / `"coverage"` / `"audit_fixture"` — direct
   mapping to the AY.2.a subtypes. `identity` is the identity tuple
   serialized as a JSON object (key: value).

5. **Identity value types**: JSON-native (str / int / float / null
   / bool). `date` values serialize as ISO-8601 strings
   (`"2030-01-02"`). The loader reverses the type-detection by
   asking each invariant's detector to round-trip its identity
   tuple shape. (Simpler alternative: never deserialize back into
   `Violation` objects; the lock test compares the LOADED JSON dict
   directly against `semantic_lock(...)` serialized through the
   same `to_json()` helper. That's the chosen shape — see "Round-
   trip" below.)

6. **Sort order is mandatory**: invariant names alphabetical;
   violations within each invariant sorted by `(kind,
   tuple(sorted(identity.items())))`; identity keys alphabetical
   within each entry. The renderer enforces; equality test relies
   on it for diffability.

### What this format does NOT do

- **No per-violation diagnostics.** The matview's extra columns
  (e.g., `drift.parent_transfer_id_min` / `parent_transfer_id_max`
  on `chain_parent_disagreement`) are NOT in the identity tuple +
  NOT in the lock. The semantic-lock contract is "did the right
  violation surface" — diagnostic columns are a separate concern
  (debug, not gate). The OLD byte locks captured them implicitly
  via SQL text.

- **No baseline-rows count.** The 90-day healthy baseline (~60k
  rows) doesn't produce violations + doesn't land in the lock.
  This is intentional — baseline shape is QS dashboard density;
  the AT.5 4-way agreement test gates that separately.

- **No row IDs / transaction.id values.** The matview's identity
  tuple keys (e.g., `transfer_id`, `transaction_id`) DO land in
  the lock when they're identity-tuple members. The OLD byte
  locks captured every `tx-NNNN` id; the semantic lock captures
  only the ones the matview surfaces.

## Round-trip + equality

### Serialize

```python
# common/spine/semantic_lock_json.py (AZ.1 lands)
def lock_to_json(
    lock: dict[str, frozenset[Violation]],
    *,
    instance: str,
    dialect: Dialect,
    canonical_anchor: date,
) -> str:
    """Serialize a semantic lock to the canonical JSON shape.
    Deterministic + byte-stable across runs."""
    violations_json: dict[str, list[dict]] = {}
    for invariant_name in sorted(lock):
        entries = []
        for v in lock[invariant_name]:
            entries.append({
                "kind": _kind_for(v),  # RuleViolation→"rule_violation" etc.
                "identity": {
                    str(k): _value_to_json(val)
                    for k, val in sorted(v.identity)
                },
            })
        # Sort entries by (kind, identity-tuple-sorted-repr).
        entries.sort(key=lambda e: (
            e["kind"],
            tuple(sorted(e["identity"].items())),
        ))
        violations_json[invariant_name] = entries
    return json.dumps(
        {
            "_comment": _COMMENT_TEXT,
            "scenario_fingerprint": {
                "instance": instance,
                "dialect": dialect.value,
                "canonical_anchor": canonical_anchor.isoformat(),
                "schema_version": 1,
            },
            "violations": violations_json,
        },
        indent=2,
        sort_keys=False,  # explicit sort done above; sort_keys=True would
                          # reorder scenario_fingerprint keys
    )
```

### Deserialize + compare

The test gate doesn't reconstruct `Violation` objects — it
serializes the live emit + compares the JSON shape directly:

```python
# tests/data/test_semantic_locks.py (AZ.2 lands)
def test_semantic_lock_matches_fresh_emit(
    locked_path, instance_name, dialect_name,
):
    fresh_lock = semantic_lock(conn, ALL_INVARIANTS)
    fresh_json = lock_to_json(fresh_lock, ...)
    on_disk = locked_path.read_text()
    assert fresh_json == on_disk, _diff(...)
```

JSON-string equality (not dict equality) so the canonical
serialization is the contract. Re-emit is byte-stable by
construction (sort + indent are fixed).

## Decisions deferred to AZ.1+

- **`schema_version` bump policy.** Start at 1; bump when the JSON
  shape changes. AZ.0 doesn't lock the bump-rules doc — happens in
  AZ.6's release notes.
- **`recon-gen data semantic-lock` CLI surface** (AZ.1) — flags,
  `--check` mode, error messages. Mirror `data lock` shape.
- **Dual-gate validation window** (AZ.3) — keep both byte lock +
  semantic lock running for N days/commits before deleting byte
  lock infra. AZ.3 picks the window.
- **`data lock` CLI retirement** (AZ.5) — once `data semantic-lock`
  replaces it, the OLD CLI command goes. AZ.5 handles the
  deprecation message vs hard-delete decision.

## Validation against AY.5's drift cases

The AY.5 re-locks shifted bytes for documented reasons:

| Drift source | Caught by byte lock? | Caught by semantic lock? |
|---|---|---|
| Per-row INSERT vs multi-row VALUES | YES (false positive) | NO ✓ (same violations either way) |
| Plant-count header text | YES (false positive) | NO ✓ (header isn't a violation) |
| `metadata.scenario_id` payload | YES (false positive) | NO ✓ (metadata isn't a violation) |
| Real violation set change | YES (correct) | YES (correct) ✓ |
| New invariant added to ALL_INVARIANTS | YES (full file diff) | YES (1-key addition) ✓ |
| Violation drops out of detection | YES | YES ✓ |

The semantic lock catches every real change + drops every false-
positive. Confirmed-fit-for-purpose.
