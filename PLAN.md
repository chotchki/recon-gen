## Phase AM - Standardize on tailwind *(active 2026-05-25)*
The HTML surface is split across two styling systems: App2 dashboards + rich-text render via Tailwind utilities (`output.css`), but the Studio editor + diagram + trainer-data pages are styled by **1,615 lines** of hand-written CSS across `editor.css` (475) + `diagram.css` (376) + `data.css` (554) + `widgets-theme.css` (210, vendored — preserved). Standardize the whole surface on Tailwind so there's ONE system. AM.0.1 audit landed 2026-05-25 (146 distinct semantic classes; see `docs/audits/am_0_1_tailwind_audit.md` once written).

**Sequenced after Phase AI deliberately** (user, 2026-05-21): the AI dogfood round-trip (structural + dashboard-equivalence, browser-driven) pins the editor's behavior + emitted HTML structure, so the CSS refactor lands with a regression safety net and doesn't churn the editor markup mid-test-build.

**AM.0 Locks (user 2026-05-25):**

- **L1 — Standardize on Tailwind defaults.** Hand-written CSS has plenty of odd numbers (`0.3125rem`, `0.4375rem`, `0.6rem`, custom rgba shadows, ad-hoc `color-mix()` blends). Snap to Tailwind's scale (`p-1`/`p-2`/`p-3`/etc.) + opacity utilities (`bg-accent/12`) + standard shadow utilities. **If the hand-built CSS is doing weird things or has odd numbers, prefer standardizing over preserving.** AM.3's screenshot verify catches any meaningful visual drift; sub-pixel adjustments don't matter.
- **L2 — Align with App2 dashboards' existing utility-only pattern.** Dashboards use raw inline utilities in `render.py` (per the existing `@source` directive scanning `render.py` / `__main__.py` / `bootstrap.js`). Studio adopts the same convention — **raw utilities, no `@apply` component classes**. The AM.0.1 audit's `@apply`-component recommendation is REJECTED in favor of one Tailwind dialect across all HTML. Highly-repeated patterns (e.g. `.entity-card` rendered 15× in editor) get a Python helper function returning the utility-string, NOT a CSS class — keeps the dialect uniform + makes class composition reviewable in Python.
  - **L2.a — Helpers return ONE utility string; variants compose at the call site.** A helper like `card_classes() -> str` is good. A helper like `card_classes(variant="editing", disabled=True) -> str` that branches internally is reinventing `@apply` in Python — the very component-class problem Tailwind exists to avoid. Compose at the call site instead: `f'{card_classes()} {"border-accent ring-2 ring-accent/15" if editing else ""}'`. If a helper grows beyond ~1 boolean param OR starts switching on enum-like state, it's a smell — break it up or move the state into the caller.
- **L3 — Bundle `data.css` into AM scope.** The AM plan as written only mentioned editor.css + diagram.css; the audit revealed `data.css` (554 lines, trainer panel) shares the same `--studio-*` token base via `diagram.css`'s `:root`. Migrating editor without data leaves trainer controls unstyled. One sweep for all three.
- **L4 — SVG stays semantic-CSS.** `.topology-svg` + `.node` + `.edge` (~49 rules) use attribute selectors (`[data-kind]`, `[data-presence]`, `[data-trainer-kinds]`) that don't translate cleanly to utility classes. Keep the SVG-specific rules in a slimmed `diagram.css` (the irreducible non-utility remainder per AM.4); migrate only the non-SVG chrome around the diagram.
- **L5 — Preserve L2 theme inheritance through Tailwind utilities.** Dashboards inherit `--color-accent` / `--color-surface` / etc. from the L2's `theme:` block via the runtime `<style>:root { --color-accent: ...; }</style>` injection in `render.py`'s page shell. Studio pages do the SAME via `_studio_routes.py::studio_theme_head(instance)` (links `/static/output.css` + injects the same per-L2 override block). Tailwind utilities like `bg-accent` / `text-accent` / `border-accent` resolve from `--color-*` at paint time, so the runtime override propagates through every utility automatically — NO new wiring needed for the migration. Migrated studio templates use the same utility set as dashboards; the inheritance is preserved by construction. Drop the `--studio-*` alias layer in `diagram.css`'s `:root` once the hand-written rules using them are gone (utilities reference `--color-*` directly).

## Phase BE - Cross-corpus duplication lint (test ↔ src), paired approaches 1+3

**Motivation (promoted from backlog AA.A.11 2026-05-24).** Every duplicated SQL string between `tests/` and `src/` is a second codebase that can pass while production breaks, or vice versa. User-flagged 2026-05-17 as "huge structural win." Paired-approach design:
- **Approach 1 — content-based AST lint.** Walk `tests/`, extract string literals with SQL fingerprints (`SELECT` / `FROM` / `<<$p` / `WHERE` / etc.) over a length threshold. For each, check if a normalized substring appears in `src/`. Flags inline SQL copies, repeated fixture strings, hand-mirrored constants. *Catches content-equality drift today.*
- **Approach 3 — provenance lint.** Same AST walker; for any test assertion that touches a SQL string OR a known production constant (sentinel value, parameter name, column name), require the value to be obtained via an `import` from `src/recon_gen/...`. Inline string literals in those assertions are the smell. *Catches future drift before it can happen.*

Land both, not either alone — they catch different failure modes and reinforce each other.

**Why exploratory** (locked 2026-05-24): unlike the no-raw-str-args lint (mechanical migration with a known shape), BE's spike produces genuine unknowns — false-positive rate at each length threshold, whether the dual approach actually reinforces or is redundant, whether structural bugs (test re-defines a constant rather than importing it) actually exist in this codebase. Either spike result is signal: real drift found → BE catches it before it ships; clean baseline → BE pins it so future drift trips loudly.

Approach 2 (token-stream tool like jscpd/PMD CPD) deliberately not pursued — both are non-Rust (Node + Java), against the standalone-binary preference. The content + provenance pair (1+3) covers the same regression class with codebase-native tooling that fits the existing `b.15.lint.*` AST infrastructure.

  the provenance rows as single-row INSERTs that skip the batcher
